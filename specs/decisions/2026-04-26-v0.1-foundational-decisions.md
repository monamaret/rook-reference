# v0.1 Foundational Decisions — Repo Structure, Config Format, Go Workspace, Admin CLI Stub

## Context and Problem Statement

Before any application code is written for v0.1 (Project Skeleton), four structural decisions must be closed. Each cascades into every subsequent release and into the CI/CD pipeline defined in the [GitHub Actions CI/CD ADR](2026-04-26-github-actions-ci-cd-pipeline.md). Deferring them risks conflicting assumptions accumulating across PRDs, ADRs, and implementation work.

The four questions:

1. Should `rook-cli`, `rook-server`, and `rook-docs` live in a single monorepo or separate repositories?
2. What format should `rook-cli`'s configuration file use?
3. How should Go modules be organised within the repository?
4. Should `rook-server-cli` (the admin binary) be stubbed and wired into the release pipeline from v0.1?

## Decision Drivers

- The CI/CD ADR assumes a monorepo with path-filtered per-component workflows — the repo topology must be confirmed before those workflow files are written
- The config format propagates to every subsequent release that reads or writes config; changing it later requires a migration
- The Go workspace strategy determines how shared code is referenced across modules and how `go.work` is maintained
- The CI/CD ADR explicitly lists `rook-server-cli` binary targets as shipping from v0.1; the entrypoint must exist for the release pipeline to be fully wired

## Decisions

### 1. Repo structure — Monorepo

**Chosen:** Single monorepo containing `rook-cli/`, `rook-server/`, and `rook-docs/` under one root.

**Rationale:** The [GitHub Actions CI/CD ADR](2026-04-26-github-actions-ci-cd-pipeline.md) is designed around a monorepo with path-filtered workflows. All Speckit artefacts (`spec.md`, `plan.md`, `tasks.md`), ADRs, and PRDs already live in this repository. Splitting components across repos would require cross-repo CI orchestration and Speckit workflow changes with no benefit at PoC scale.

**Consequences:**
- Good, because CI path filters keep component builds independent without separate repos.
- Good, because ADRs, specs, and code are co-located and versioned together.
- Bad, because a future split (e.g., open-sourcing `rook-cli` independently) would require repository surgery — acceptable given PoC scope.

**Rejected alternative:** Multi-repo (one repo per component). Rejected because it adds cross-repo CI complexity and no concrete benefit at current team and service count.

---

### 2. Config file format — JSON

**Chosen:** `rook-cli` stores its configuration as JSON at `$XDG_CONFIG_HOME/rook/config.json`.

**Rationale:** JSON is supported by the Go standard library (`encoding/json`) with no additional dependencies. PRD001 already references `config.json` by name. The config file is written by `rook-cli` internally (not hand-edited by users in normal operation), so the absence of comment support is not a practical drawback for the PoC.

**Consequences:**
- Good, because zero external dependencies for config parsing.
- Good, because consistent with PRD001's explicit reference to `config.json`.
- Good, because JSON is unambiguous — no TOML multiline/array syntax edge cases, no YAML indentation hazards.
- Bad, because users who want to inspect or manually edit the config file cannot add inline comments.
- Neutral, because a future migration to a comment-supporting format (TOML) is straightforward — the config schema is small and the read/write utilities are the only touch point.

**Rejected alternatives:**
- **TOML** — good Go ecosystem support (`BurntSushi/toml`), but adds a dependency and the comment benefit is irrelevant if the file is machine-written.
- **YAML** — familiar in ops contexts but complex to parse correctly (multiple spec versions, implicit type coercions, indentation sensitivity). Not worth the complexity for a small config file.

### 3. Go workspace strategy — `go.work` with per-component `go.mod`

**Chosen:** A single `go.work` file at the repository root. Each deployable component (`rook-cli/`, `rook-server/`, `rook-docs/` if applicable) maintains its own `go.mod` with its own module path and dependency graph.

**Rationale:** Go workspaces (`go.work`) are the idiomatic solution for monorepos containing multiple Go modules. They allow local module references without `replace` directives in `go.mod`, keeping each component's module graph clean and independently releasable. Each component can pin its own Go version and dependency set.

**Structure:**
```
go.work                        # workspace root — lists all Go modules
rook-cli/
  go.mod                       # module: github.com/rook-project/rook-reference/rook-cli
rook-server/
  go.mod                       # module: github.com/rook-project/rook-reference/rook-server
```

`go.work` is committed to the repository. `go.work.sum` is also committed to ensure reproducible builds across contributors and CI.

**Consequences:**
- Good, because each module has an independent dependency graph — `rook-cli` dependencies do not pollute `rook-server` and vice versa.
- Good, because `go.work` local references replace `replace` directives, keeping `go.mod` files clean.
- Good, because CI can run `go test ./...` scoped to a single module's directory without workspace-level side effects.
- Bad, because `go.work` must be updated when new Go modules are added to the repo.
- Neutral, because `go.work` is a local/CI-only mechanism — it does not affect how modules are imported by external consumers.

**Rejected alternative:** Single `go.mod` for the entire repo. Rejected because it couples `rook-cli` and `rook-server` into a single version graph, prevents independent binary compilation scoping, and makes it harder to reason about which dependencies each deliverable actually requires.

---

### 4. `rook-server-cli` stub in v0.1 — Yes

**Chosen:** `rook-server/cmd/admin/main.go` is created in v0.1 as a stub binary that prints version information and exits 0.

**Rationale:** The [GitHub Actions CI/CD ADR](2026-04-26-github-actions-ci-cd-pipeline.md) explicitly lists `rook-server-cli` (linux/amd64, darwin/arm64) as a release binary produced from v0.1. If the entrypoint does not exist, the `release.yml` binary job fails from the first tag. A no-op stub satisfies the pipeline requirement while deferring admin functionality to v0.3+.

**Consequences:**
- Good, because `release.yml` is fully wired and verified from v0.1 — no pipeline gaps to discover later.
- Good, because the stub establishes the canonical entrypoint path (`rook-server/cmd/admin/`) that all subsequent admin CLI work will build on.
- Bad, because the binary shipped in the v0.1 GitHub Release does nothing useful — acceptable given v0.1 is a developer-facing skeleton release.

**Rejected alternative:** Omit the stub and skip `rook-server-cli` from the v0.1 release pipeline. Rejected because it leaves an untested gap in `release.yml` that will fail silently until a later release attempts to ship the binary.

## Implementation Notes

- `go.work` must list both `rook-cli/` and `rook-server/` from the start; add new modules here as they are introduced.
- Go version in each `go.mod` should be pinned to the same minor version (e.g., `go 1.23`) and kept in sync across modules via a Makefile target or contributor guideline.
- `rook-server/cmd/admin/main.go` stub should print: `rook-server-cli version <version>` and exit 0. Version is injected at build time via `-ldflags`.
- Config file path resolution must respect `XDG_CONFIG_HOME` with a fallback to `~/.config/rook/config.json` on Linux/macOS and the documented WSL equivalent.
