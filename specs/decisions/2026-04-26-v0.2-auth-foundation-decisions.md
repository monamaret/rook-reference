# v0.2 Auth Foundation Decisions — Session Token Format, Nonce Lifecycle, SSH Key Selection

## Context and Problem Statement

PRD003 (v0.2 — Auth Foundation) listed three open questions that must be resolved before the challenge-response auth flow can be specified and implemented:

1. Should the session token be a signed JWT or an opaque random token?
2. What is the nonce TTL, and how are expired nonces cleaned up in Firestore?
3. When `rook-cli` encounters multiple SSH keys in `~/.ssh/`, how does it select which key to use?

The [SSH Auth Identity Chain and Cloud Run Topology ADR](2026-04-25-ssh-auth-identity-chain-and-cloud-run-topology.md) resolves questions 1 and 2 as part of its chosen option. Question 3 is left open in that ADR. This ADR closes all three, consolidates the session token and nonce decisions from the SSH Auth ADR for PRD003 traceability, and adds the key selection decision.

## Decision Drivers

- Session credential must be stateless from the service's perspective at PoC scale — services should not need to decode or validate claims locally
- Nonce cleanup must be bounded without requiring application-level scheduled jobs or manual maintenance
- SSH key selection UX must be discoverable for new users who have not previously configured a preferred key

## Decisions

### 1. Session token format — Opaque random token

**Chosen:** Session tokens are cryptographically random 32-byte values, hex-encoded. They carry no embedded claims. All session data (user ID, group, expiry) is stored server-side in Firestore under `sessions/{token}`.

**Rationale:** Established by the [SSH Auth ADR](2026-04-25-ssh-auth-identity-chain-and-cloud-run-topology.md) (Option B). Session validation is performed by `user-service.ValidateSession(token, space_id)` via gRPC — services never decode the token locally. Opaque tokens are trivially revocable: deleting the Firestore document immediately invalidates the token on the next request. JWTs would require either a blocklist (adds state) or short expiry with refresh logic (adds complexity) to support revocation.

**Token properties:**

| Property | Value |
|----------|-------|
| Format | Random 32-byte value, hex-encoded |
| Storage (server) | Firestore `sessions/{token}` — user ID, group, space ID, expiry |
| Storage (CLI) | In-memory only; discarded on process exit |
| TTL | 1 hour |
| Revocation | Delete the Firestore document; next request returns `401` |

**Consequences:**
- Good, because revocation is immediate — no blocklist or JWT-specific infrastructure required.
- Good, because services never decode session data locally — `ValidateSession` is the single source of truth.
- Good, because token generation is trivial (`crypto/rand` from the Go standard library).
- Bad, because every authenticated request incurs a `ValidateSession` gRPC round-trip to `user-service` — mitigable with a short in-process cache (TTL: 5 minutes) if latency becomes a concern.
- Neutral, because session state lives in Firestore — consistent with the project's data layer decision.

**Rejected alternative:** Signed JWT. Rejected because JWTs complicate revocation (require a blocklist or accept a revocation lag bounded by token TTL), introduce a signing key that must be provisioned and rotated across all services, and add claims-decoding logic that must be kept in sync with the server-side schema.

### 2. Nonce lifecycle — 60-second TTL, Firestore TTL policy cleanup

**Chosen:** Nonces are stored in Firestore under `auth/nonces/{nonce}` with an `expires_at` timestamp 60 seconds from issuance. Application code enforces the validity window by checking `expires_at` at read time. A Firestore TTL policy configured on the `expires_at` field automatically deletes expired documents within 72 hours — no Cloud Scheduler job or on-read purge is required. Nonces are marked used on first successful verification to prevent replay regardless of TTL.

**Rationale:** Established by the [SSH Auth ADR](2026-04-25-ssh-auth-identity-chain-and-cloud-run-topology.md). The 60-second TTL is tight enough to limit nonce exposure while giving the user adequate time to complete the sign-and-verify round-trip. Firestore TTL policies are a zero-ops cleanup mechanism — they require no additional infrastructure and are consistent with the operational simplicity driver. Application-level expiry enforcement ensures correctness regardless of Firestore TTL propagation lag.

**Nonce properties:**

| Property | Value |
|----------|-------|
| Storage | Firestore `auth/nonces/{nonce}` |
| Size | 32 bytes, hex-encoded |
| Validity window | 60 seconds (enforced at read time via `expires_at`) |
| Cleanup | Firestore TTL policy on `expires_at` (auto-delete within 72h of expiry) |
| Replay prevention | Marked used atomically on first successful verification; second use returns `401` |

**Consequences:**
- Good, because no scheduled job or background process is required — Firestore TTL handles collection hygiene.
- Good, because the 60-second window is consistent with PRD001's specification.
- Good, because replay prevention is independent of TTL — a nonce used once is invalidated immediately, regardless of whether Firestore has yet deleted it.
- Neutral, because Firestore TTL deletion can lag up to 72 hours after expiry — the collection may transiently contain expired nonces, but application code never accepts them.

**Rejected alternatives:**
- **Cloud Scheduler + Cloud Function purge**: operationally heavier, adds a new GCP resource to manage, and provides no correctness benefit over Firestore TTL policies.
- **On-read purge**: purging expired nonces at read time adds write-on-read behaviour that complicates handler logic and creates race conditions under concurrent auth attempts.

### 3. SSH key selection — Prompt at auth time

**Chosen:** When `rook auth` is invoked and multiple SSH private keys are present in `~/.ssh/`, the CLI lists the available keys (by filename and key type) and prompts the user to select one interactively via a Bubble Tea list component. The selected key path is remembered in `config.json` for the remainder of the session but is not persisted across process exits — the user is prompted again on the next `rook auth` invocation.

**Rationale:** For a PoC with a small user base, discoverability matters more than convenience. A prompt surfaces available keys explicitly, reducing the chance of auth failure caused by selecting the wrong key (e.g., a key whose public key is not registered with the server). Config-file-only selection would require the user to know the correct path before first auth, which is a poor first-run experience. Prompting is consistent with the fully terminal-native, TUI-first design established in the rook-cli UX ADR.

**Key discovery rules:**
- Scan `~/.ssh/` for files matching `id_*` without a `.pub` extension (standard key file naming).
- Exclude known non-key files (`known_hosts`, `authorized_keys`, `config`).
- Display filename, key type (parsed from the corresponding `.pub` file if present), and key fingerprint.
- If only one key is found, skip the prompt and use it directly (with an informational message).
- If no keys are found, display an error and direct the user to generate a key or specify a path via `--key` flag.

**CLI flag:** `rook auth --key <path>` allows bypassing the prompt for scripted or non-interactive use (e.g., CI environments or admin automation).

**Consequences:**
- Good, because new users immediately see which keys are available without needing to know `~/.ssh/` contents in advance.
- Good, because the prompt prevents silent selection of an unregistered key.
- Good, because a `--key` flag preserves non-interactive use.
- Bad, because repeat users are prompted on every `rook auth` invocation — mitigated by the single-key fast path and the low frequency of re-auth in normal use (session TTL: 1 hour).
- Neutral, because the selected key is not persisted — if this proves friction in practice, persisting the last-used key path to `config.json` is a straightforward follow-up in a later release.

**Rejected alternatives:**
- **Config file only**: requires the user to know the correct key path before first auth; poor first-run experience; does not surface available options.
- **`ssh-agent` first, config fallback**: adds agent detection complexity and creates an implicit selection path that is hard to debug when auth fails because the agent loaded the wrong key.

## Implementation Notes

- Session token generation: `crypto/rand.Read` into a 32-byte slice, `hex.EncodeToString`. No external dependency.
- Nonce generation: same as session token — `crypto/rand.Read`, 32 bytes, hex-encoded.
- Firestore TTL policy must be created on the `expires_at` field of the `auth/nonces` collection before the first deployment. Verify with: `gcloud firestore fields ttls describe --collection-group=nonces --field-path=expires_at`.
- SSH key discovery in `rook-cli/auth/keys.go`: use `os.ReadDir("~/.ssh/")`, filter by naming convention, parse key type from `.pub` sibling if present.
- Key selection prompt: Bubble Tea `list.Model` component, consistent with the launcher home screen pattern established in v0.3.
- `rook auth --key <path>`: accepted as a `cobra` flag; bypasses discovery and prompt entirely.
- Session token must never be written to disk by `rook-cli` — in-memory only (`context.Context` or a package-level auth state struct for the process lifetime).
