# Use gRPC for rook-server Inter-Service Communication

## Context and Problem Statement

`rook-server` is structured as a set of independent Go binaries, each deployed as its own Cloud Run service with its own Firestore collection namespace. The services must coordinate — most commonly, services need to verify SSH-key identity, resolve space membership, and check group ACL against the user management service before processing a request. What protocol should govern these inter-service calls?

This decision was identified as a hard prerequisite in [`2026-04-25-rook-reference-system-architecture.md`](2026-04-25-rook-reference-system-architecture.md) (open question 1). No service implementation may begin until it is resolved.

## Decision Drivers

- **Schema safety**: the system will serve ~10 real users over several weeks; silent interface drift between services is a real operational risk even at this scale
- **Cloud Run compatibility**: each service exposes a single port; the protocol must work within that constraint
- **Go ecosystem**: Go's gRPC and Protobuf support is mature and well-documented
- **Solo developer**: toolchain cost is a one-time setup investment, not an ongoing burden
- **PoC scope**: no Kubernetes, no service mesh; the protocol must not require mesh infrastructure to function

## Considered Options

- gRPC with Protobuf
- REST/HTTP with JSON

## Decision Outcome

Chosen option: **gRPC with Protobuf**, because Protobuf `.proto` files define a typed, versioned contract between services that is enforced at compile time. With a real deployment over several weeks, compile-time schema safety is more valuable than the marginal simplicity of plain HTTP. Cloud Run supports HTTP/2 (and therefore gRPC) natively — no additional infrastructure is required.

### Consequences

- Good, because Protobuf schemas are the source of truth for inter-service contracts — drift is caught at compile time, not at runtime
- Good, because Cloud Run supports gRPC over HTTP/2 natively; no proxy or sidecar required
- Good, because generated Go stubs eliminate hand-written HTTP client/server boilerplate
- Good, because `.proto` files serve as machine-readable documentation of each service's interface
- Bad, because `protoc` and the Go gRPC plugins (`protoc-gen-go`, `protoc-gen-go-grpc`) must be installed and maintained as part of the developer toolchain
- Bad, because inter-service calls are not directly inspectable with `curl` — `grpcurl` or a similar tool is needed for ad-hoc debugging
- Neutral, because gRPC streaming is available but not planned for initial services; all RPCs are unary

## Implementation Plan

### Affected paths

- `rook-server/proto/` — all `.proto` definitions (create from scratch)
- `rook-server/gen/go/` — generated Go stubs (generated by `buf generate`, committed)
- `rook-server/user-service/` — gRPC server implementation for `UserService`
- `rook-server/messaging-service/` — gRPC client consuming `UserService` stubs
- `rook-server/guides-service/` — gRPC client consuming `UserService` stubs
- `rook-server/stash-service/` — gRPC client consuming `UserService` stubs

### Dependencies

```

```

Developer toolchain (not Go module deps):

```
github.com/bufbuild/buf         # proto linting, breaking change detection, code generation
protoc-gen-go                   # Go Protobuf plugin

grpcurl                         # ad-hoc debugging (optional but recommended)
```

### Patterns to follow

- **Package naming**: use versioned packages — `rook.user.v1`, `rook.messaging.v1`, etc. This allows non-breaking evolution.
- **Service naming**: one `.proto` file per service; one `service` block per file.
- **Unary RPCs only** for the initial implementation — no streaming.
- **Error model**: use standard gRPC status codes (`codes.NotFound`, `codes.PermissionDenied`, etc.) — do not invent a bespoke error envelope.
- **Cloud Run service-to-service auth**: each gRPC client must attach a GCP OIDC token as a `Authorization: Bearer` header. Use the Cloud Run metadata server to fetch tokens at call time. Do not hardcode credentials.
- **Generated stubs**: run `buf generate` from `rook-server/proto/` to regenerate `gen/go/`. Commit generated files — do not gitignore them. This ensures any developer can build without running codegen.

### Patterns to avoid

- Do not make raw HTTP calls between services — all inter-service communication must use generated gRPC stubs
- Do not define RPCs that cross service ownership boundaries (e.g., messaging service defining RPCs that belong in user management)
- Do not use gRPC streaming for the initial implementation — add it in a follow-up ADR if needed
- Do not expose gRPC endpoints to external clients (the CLI) — external-facing interfaces are defined separately per service (SSH via `charmbracelet/wish` for auth, HTTP for data sync)

### Initial UserService proto (dominant inter-service contract)

The user management service is the only service all others depend on. Its proto should be defined first:

```protobuf
syntax = "proto3";

package rook.user.v1;

option go_package = "github.com/rook-project/rook-reference/rook-server/gen/go/user/v1;userv1";

service UserService {
  // Resolve a user by SSH public key fingerprint. Returns the user's identity.
  rpc GetUserByKey(GetUserByKeyRequest) returns (GetUserByKeyResponse);

  // Check whether a user is a member of a given space and return their group.
  rpc GetSpaceMembership(GetSpaceMembershipRequest) returns (GetSpaceMembershipResponse);

  // Check whether a user's group has access to a given app in a space.
  rpc CheckAppAccess(CheckAppAccessRequest) returns (CheckAppAccessResponse);
}
```

Full message definitions to be filled in during service implementation. The three RPCs above cover the dominant inter-service call patterns identified in the architecture.

### Configuration

Each service binary reads the following environment variables for gRPC client configuration:

| Variable | Purpose |
|----------|---------|
| `USER_SERVICE_ADDR` | gRPC address of the user management service (e.g., `user-service-xyz-uc.a.run.app:443`) |
| `MESSAGING_SERVICE_ADDR` | gRPC address of the messaging service |
| `GUIDES_SERVICE_ADDR` | gRPC address of the guides service |
| `STASH_SERVICE_ADDR` | gRPC address of the stash service |

All addresses are environment-variable-driven — no hardcoded service URLs anywhere in the codebase (see system architecture non-goals).

TLS is handled by Cloud Run's managed certificates. Clients connect with `grpc.WithTransportCredentials(credentials.NewTLS(&tls.Config{}))` — do not use `grpc.WithInsecure()` in any deployed service.

## Pros and Cons of the Options

### gRPC with Protobuf

- Good, because Protobuf schemas are the authoritative contract — compile-time safety across all services
- Good, because Cloud Run supports HTTP/2 and gRPC natively
- Good, because generated stubs eliminate boilerplate and make interface changes visible immediately
- Good, because `buf` provides linting and breaking-change detection
- Bad, because `protoc`/`buf` toolchain adds developer setup complexity
- Bad, because inter-service traffic is not human-readable without tooling (`grpcurl`)

### REST/HTTP with JSON

- Good, because Go's `net/http` stdlib is sufficient — no additional dependencies or toolchain
- Good, because requests are directly inspectable with `curl`
- Bad, because no enforced schema contract — interface drift between services is a silent runtime failure
- Bad, because hand-written HTTP client/server code across four services is significant boilerplate
- Bad, because versioning and contract evolution require manual discipline (OpenAPI or similar)
