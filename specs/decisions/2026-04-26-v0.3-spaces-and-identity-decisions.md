# v0.3 Spaces and Identity Decisions — Group Cardinality, App Tile Availability, Zero-Space State

## Context and Problem Statement

PRD004 (v0.3 — Spaces and Identity) listed three open questions that must be resolved before the Spaces data model and the launcher home screen TUI can be specified and implemented:

1. Are groups space-scoped or cross-space? This determines the Firestore schema for groups and the shape of every downstream ACL query.
2. Should app tile availability in the launcher be driven by the space config (static) or by live service reachability (dynamic)?
3. What does `rook-cli` show when a user authenticates successfully but belongs to zero spaces?

None of these questions are resolved by existing ADRs. The [rook-cli UX ADR](2026-04-25-rook-cli-features-and-ux-architecture.md) establishes the launcher layout and the space selector behaviour (hidden for single-space users) but does not address cardinality, tile sourcing, or the zero-space edge case. This ADR closes all three.

## Decision Drivers

- Firestore document structure for groups must be settled before any collection schema or query code is written — restructuring after the fact cascades through every service that reads group membership
- The launcher home screen is the primary UX surface and sets the Bubble Tea model pattern for all subsequent screens; its data sources must be unambiguous before implementation begins
- The zero-space state is a real edge case at PoC scale (admin forgets to assign a new user to a space); a clear, recoverable UX is required that doesn't confuse the user or destroy their config

## Decisions

### 1. Groups are space-scoped

**Chosen:** A group exists within exactly one space. A user may belong to different groups in different spaces (e.g., `editors` in space A, `readers` in space B), but a single group document cannot be assigned to multiple spaces. There are no server-level or cross-space groups in the PoC.

**Firestore schema:** Groups are a sub-collection of their parent space:

```
spaces/{space-id}/
  groups/{group-id}/
    displayName: string
    description: string
    apps: []string          # list of app IDs this group can access (ACL)

spaces/{space-id}/members/{user-id}/
  groupId: string           # the group this user belongs to within this space
  joinedAt: timestamp
```

`user-service` resolves a user's group for a given space by reading `spaces/{space-id}/members/{user-id}.groupId` and then fetching `spaces/{space-id}/groups/{group-id}` for the ACL. Both reads are scoped to the space — no cross-collection query is required.

**Rationale:** Space-scoped groups are the simplest model that satisfies the PoC requirement: ACL enforcement per space with no cross-space data access. A user's permissions in one space are entirely independent of their permissions in another. Firestore document paths make the scoping explicit and enforce it structurally — there is no query that can accidentally return group data from a different space.

**Consequences:**
- Good, because the schema is flat and readable — `spaces/{space-id}/groups/{group-id}` is self-documenting.
- Good, because ACL resolution is two Firestore reads, both scoped to the same space — no joins, no cross-collection fan-out.
- Good, because adding a user to a new space with a different group requires only a new `members/{user-id}` document — no mutation of a shared group entity.
- Bad, because a server admin cannot define a group once and reuse it across spaces — groups must be created per-space. Acceptable at PoC scale with a small number of spaces.
- Neutral, because cross-space groups can be introduced in a later milestone by promoting groups to a top-level collection — the current schema does not foreclose that migration.

**Rejected alternative:** Cross-space groups (server-level group entity assigned to multiple spaces). Rejected because it requires a join between a top-level `groups` collection and a per-space membership record on every ACL check, and it introduces a shared mutable entity that complicates space data segregation — a core architectural invariant.

### 2. App tile availability — Static list from space config

**Chosen:** The launcher renders app tiles based solely on the list returned by `GET /spaces/{space-id}/apps`, which reflects the group's ACL for that space as stored in Firestore. The launcher does not probe service endpoints to determine tile availability. An app tile is shown if and only if the user's group has ACL permission for it; it is greyed out (not hidden) if the server is unreachable when the launcher loads.

**Tile state model:**

| Condition | Tile state |
|-----------|-----------|
| Group has ACL permission + server reachable | Active (selectable) |
| Group has ACL permission + server unreachable | Greyed out (shown, not selectable) |
| Group does not have ACL permission | Hidden |

Reachability is determined by the existing server status poll at launch (one ping per configured server, non-blocking) — not a per-service probe. All app tiles for a given server are greyed out together if that server's status check fails.

**Rationale:** Dynamic per-service probing adds latency, complexity, and a new failure mode: a service that is temporarily slow could cause its tile to flicker or disappear. For a PoC, the ACL list is the authoritative source of what a user is allowed to access. Service availability is an operational concern — the launcher reflects permissions, not infrastructure health. The server status pane already communicates reachability; duplicating that signal at the tile level is redundant.

**Consequences:**
- Good, because tile rendering requires only the data already fetched by `GET /spaces/{id}/apps` — no additional network calls.
- Good, because tile set is stable across the launcher session — no flickering from intermittent service probes.
- Good, because consistent with the offline-first principle: the launcher renders from cached ACL data when the server is unreachable, showing the correct tile set in a greyed-out state.
- Bad, because a tile may appear active for a service that is deployed but broken — the user discovers the failure only on navigation, not on the launcher. Acceptable at PoC scale.
- Neutral, because per-service health signals can be added to the tile model in a later release without changing the ACL-driven tile set logic.

**Rejected alternative:** Dynamic tile availability based on live service reachability. Rejected because it adds per-service HTTP probes to the launcher startup path, introduces latency variability, and conflates permission (ACL) with availability (infrastructure health) in the UI.

### 3. Zero-space user state — Targeted error screen

**Chosen:** If `GET /spaces` returns an empty list after successful authentication, `rook-cli` renders a dedicated full-screen error state (not the first-run setup flow) with a clear message and actionable guidance:

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  You are not a member of any spaces.                    │
│                                                         │
│  Your identity was verified, but your account has not   │
│  been added to a space yet.                             │
│                                                         │
│  Ask your server admin to run:                          │
│  rook-server-cli user add-to-space --user <you>         │
│                   --space <space-id>                    │
│                                                         │
│  Press R to retry  ·  Press Q to quit                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

Pressing `R` re-fetches `GET /spaces` without re-authenticating. Pressing `Q` exits. The first-run setup flow is not re-triggered — it runs only when `config.json` is absent.

**Rationale:** A user who reaches the zero-space screen has already completed first-run setup (config exists) and authenticated successfully (session token issued). Re-running the setup flow would re-prompt for server address and storage directory — information the user has already provided correctly. The problem is not misconfiguration; it is a pending admin action. A targeted screen that names the exact admin command to run is faster to resolve and less confusing than a setup re-run that offers no path to resolution.

**Consequences:**
- Good, because the screen surfaces the exact admin command needed, giving the user something actionable without guessing.
- Good, because existing config is not modified or re-prompted — the user's setup is intact.
- Good, because the retry path (press `R`) allows resolution without re-launching `rook-cli`.
- Bad, because a user who genuinely misconfigured their server address during setup will see this screen and may be confused — they would need to edit `config.json` manually. Acceptable at PoC scale; a `rook config edit` command can be added in a later release.
- Neutral, because the screen is a terminal dead-end (retry or quit) — no navigation to other launcher features. This is intentional: without space membership, no launcher features are accessible anyway.

**Rejected alternative:** Re-run the full first-run setup flow. Rejected because the setup flow prompts for server address and storage directory — both of which are already correctly configured — and offers no mechanism to add the user to a space. It would confuse the user into thinking they need to reconfigure their client when the required action is a server-side admin operation.

## Implementation Notes

- **Firestore collection creation:** `spaces`, `spaces/{id}/groups`, and `spaces/{id}/members` collections must be provisioned (or emerge naturally from `rook-server-cli space create` and `user add-to-space`) before any `GET /spaces` call can succeed.
- **`GET /spaces/{id}/apps` response:** Returns the list of app IDs from `spaces/{space-id}/groups/{group-id}.apps` for the authenticated user's group. The `user-service` handler resolves the group from `members/{user-id}.groupId` before fetching the ACL list — two Firestore reads, both within the same space path.
- **ACL cache:** The result of `GET /spaces` and `GET /spaces/{id}/apps` is cached to `<storage-dir>/cache/spaces.json` for offline launcher rendering. Cache is invalidated on session change or explicit refresh (manual `R` key in the launcher).
- **Launcher tile rendering:** Tile visibility is derived from the cached ACL list. Tile active/greyed state is derived from the server status poll result already present in the launcher model — no additional state is required.
- **Zero-space screen:** Implemented as a distinct Bubble Tea model returned by the launcher's `Update` function when `GET /spaces` succeeds with an empty list. It is not a modal overlay — it takes over the full terminal until the user retries or quits.
- **`displayName` field:** Both `spaces/{id}` and `spaces/{id}/groups/{id}` documents must include a `displayName` string field. The launcher and space selector render this directly — no secondary lookup required.
