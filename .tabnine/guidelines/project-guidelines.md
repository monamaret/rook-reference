# Project Guidelines

## 1. Speckit Workflow

All feature development in this project follows the **Speckit Full SDD Cycle**. Every new feature must pass through each stage in order, with human review gates between stages. Do not skip or reorder stages.

### Stages (in order)

| Step | Command | Description |
|------|---------|-------------|
| 1 | `/speckit.specify` | Generate the feature specification (spec.md) |
| — | **Review gate** | Human approves the spec before proceeding |
| 2 | `/speckit.plan` | Produce the implementation plan, research, data model, and contracts |
| — | **Review gate** | Human approves the plan before proceeding |
| 3 | `/speckit.tasks` | Generate a dependency-ordered `tasks.md` |
| 4 | `/speckit.taskstoissues` | Sync every task in `tasks.md` to a GitHub Issue |
| — | **Review gate** | Human reviews the created issues before implementation begins |
| 5 | `/speckit.implement` | Execute the tasks defined in `tasks.md` |

**Never begin a later stage until the prior review gate has been explicitly approved.**

### Phase conventions

- **specify** — Describes *what* to build (user stories, acceptance criteria).
- **plan** — Describes *how* to build it (tech stack, architecture, data model, API contracts, research).  
  Output artifacts: `plan.md`, `research.md`, `data-model.md`, `contracts/`, `quickstart.md`.
- **tasks** — Breaks the plan into an ordered, dependency-aware checklist (`tasks.md`).  
  Task IDs follow the format `T001`, `T002`, …; parallel tasks are marked `[P]`; user-story tasks carry a `[US1]` / `[US2]` … label.
- **implement** — Executes tasks phase by phase, respecting TDD order (test tasks before implementation tasks), and marks each completed task as `[X]` in `tasks.md`.

## 2. Git Branching

The project uses the **Speckit Git Branching Workflow extension** (`speckit.git.*`). Adhere to these rules at all times.

### 2.1 Repository initialisation

Before any constitution or spec work begins, the repository must be initialised via `/speckit.git.initialize`. This is a mandatory, automatic pre-hook (`before_constitution`).

### 2.2 Feature branches

A dedicated feature branch **must** be created before `speckit.specify` runs. This is a mandatory automatic pre-hook (`before_specify`). Branch names are generated by the extension script — do not create them by hand.

**Branch naming** is controlled by `branch_numbering` in `.specify/extensions/git/git-config.yml`:

| Mode | Example |
|------|---------|
| `sequential` (default) | `003-add-user-auth` |
| `timestamp` | `20260319-143022-add-user-auth` |

Rules:
- Short name: 2–4 words in action-noun format (e.g., `add-user-auth`, `fix-payment-bug`).
- Preserve technical acronyms as-is (e.g., `OAuth2`, `gRPC`, `JWT`).
- Never pass `--number` to the creation script; the script determines the number automatically.
- Run the creation script exactly **once** per feature.
- Use the Bash script on Linux/macOS: `.specify/extensions/git/scripts/bash/create-new-feature.sh --json --short-name "<short-name>" "<feature description>"`

### 2.3 Commits during the workflow

Between stages, use `/speckit.git.commit` (the auto-commit hook) to stage and commit Speckit-generated artefacts. The hook configuration lives in `.specify/extensions/git/git-config.yml` under `auto_commit`.

| Hook event | When it runs |
|------------|--------------|
| `before_clarify` / `after_clarify` | Around spec clarification |
| `before_plan` / `after_plan` | Around planning |
| `before_tasks` / `after_tasks` | Around task generation |
| `before_implement` / `after_implement` | Around implementation |

Commit messages for Speckit artefacts follow the per-event `message` in `git-config.yml`. If no message is set, the default `[Spec Kit] …` prefix is used.

### 2.4 Branch validation

Always verify the active branch conforms to naming conventions before starting implementation work:

```bash
/speckit.git.validate
```

If the current branch does not conform, create a properly named branch before continuing.

## 3. GitHub Tasks for Speckit Tasks

Every task entry in `tasks.md` must have a corresponding **GitHub Issue** in the repository before implementation of that task begins. This ensures full traceability from plan → issue → PR → code.

### 3.1 Creating issues from tasks

Use `/speckit.taskstoissues` to sync `tasks.md` tasks to GitHub Issues automatically. Run this after `/speckit.tasks` generates the task list and before `/speckit.implement` starts.

Issue titles should mirror the task description exactly. The issue body must include:
- The task ID (e.g., `T012`)
- The user-story label (e.g., `US1`) if applicable
- Whether the task is parallelisable (`[P]`)
- The target file path(s) stated in the task

Label issues consistently:
- `phase/setup`, `phase/foundational`, `phase/polish` for non-story tasks
- `story/us1`, `story/us2`, … for user-story tasks
- `parallel` for `[P]` tasks

### 3.2 Closing issues

A GitHub Issue is closed **only** when the corresponding task is marked `[X]` in `tasks.md` **and** its implementing PR has been merged.

## 4. Pull Requests for Implementation

Each task (or tightly related group of tasks within the same phase) must be implemented and delivered in its own **Pull Request**. Do not bundle multiple unrelated tasks into a single PR.

### 4.1 PR scope rules

- One PR per task is the default. Tasks within the same phase that operate on *completely distinct* files and have no shared dependencies may optionally be grouped into one PR if they are marked `[P]` (parallel).
- Never merge implementation work directly to `main`/`master`. All code must go through a PR from the feature branch (or a task sub-branch off the feature branch).
- PRs targeting the feature branch are merged as the feature progresses. The feature branch itself is merged to `main` via a final PR once all tasks are complete and the checklist passes.

### 4.2 PR description template

Each PR description must include:

```
## Summary
<!-- What does this PR implement? Reference the task ID and linked issue. -->

## Tasks completed
- [ ] T00X – <description>

## Related issues
Closes #<issue-number>

## Testing
<!-- How was this tested? Which test files were added/modified? -->

## Checklist
- [ ] All task checkboxes marked [X] in tasks.md
- [ ] Tests added/updated
- [ ] No secrets or credentials committed
- [ ] Follows project conventions and constitution
```

### 4.3 Branch strategy for task PRs

When a feature has many tasks, create short-lived **task branches** off the feature branch:

```
main
 └─ 003-add-user-auth          ← feature branch (created by speckit.git.feature)
     ├─ 003-add-user-auth/T001-project-setup
     ├─ 003-add-user-auth/T005-user-model
     └─ 003-add-user-auth/T012-auth-middleware
```

Each task branch is PR'd back into the feature branch. Once all task PRs are merged, the feature branch is PR'd into `main`.

## 5. Commit Message Conventions

Follow Conventional Commits for all hand-crafted commits (non-Speckit-hook commits):

```
<type>(<scope>): <short summary>

[optional body]

[optional footer — Refs #<issue>, Closes #<issue>]
```

Common types: `feat`, `fix`, `test`, `docs`, `chore`, `refactor`, `ci`.

Speckit auto-commits use the `[Spec Kit]` prefix defined in `git-config.yml` and are exempt from the Conventional Commits format.

## 6. TABNINE.md Plan Reference

The `TABNINE.md` file contains a `<!-- SPECKIT START --> … <!-- SPECKIT END -->` block. This block **must** always point to the current plan file for the active feature. It is updated automatically by `/speckit.plan` (Phase 1, step 3). Do not edit this block manually.

## 7. Agent Plan Mode

When the agent enters **plan mode** (e.g., to research, design, or map out a complex change before implementation), it may write plan files to:

```
.tabnine/agent/plans/
```

Use this directory freely to capture intermediate thinking, design sketches, spike notes, architectural proposals, or any structured reasoning that should persist across turns. Files here are working documents — they are not formal Speckit artefacts and do not replace `plan.md`, `research.md`, or `tasks.md`.

### Conventions

- **File naming**: use a short descriptive name with a date prefix, e.g. `2026-04-26-auth-spike.md`, `2026-04-26-grpc-design.md`.
- **Format**: Markdown. No fixed schema required — use whatever structure best captures the plan (headings, decision tables, pseudocode, diagrams in text, etc.).
- **Scope**: one file per distinct planning topic. Do not concatenate unrelated plans into a single file.
- **Lifecycle**: plan files may be committed alongside feature work for traceability, but they are never required to be reviewed or approved as part of the Speckit gate process.

## 8. Summary: Workflow Checklist

Before starting any new feature, confirm:

- [ ] Feature branch created via `/speckit.git.feature` (or the automatic `before_specify` hook)
- [ ] `spec.md` generated and approved
- [ ] `plan.md` (+ `research.md`, `data-model.md`, `contracts/`) generated and approved
- [ ] `tasks.md` generated
- [ ] GitHub Issues created for all tasks (via `/speckit.taskstoissues`)
- [ ] Each task implemented in its own PR, linked to its issue
- [ ] `tasks.md` task marked `[X]` and issue closed when PR is merged
- [ ] All checklists passing before final feature-branch PR to `main`
